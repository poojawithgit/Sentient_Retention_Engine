# Sentient Retention Engine - Code Review and Optimization Plan

## Executive Summary
This plan outlines identified issues and recommended improvements for the Sentient Retention Engine codebase. The analysis covers logical errors, edge-case failures, performance bottlenecks, and security concerns across backend, frontend, and database layers.

## Critical Issues Requiring Immediate Attention

### 1. WebSocket Broadcasting Inefficiency (Backend)
**Location**: `backend/src/app.js` lines 68-84 (broadcast function)
**Problem**: Current implementation broadcasts to ALL connected clients for every event, regardless of relevance. This doesn't scale.
**Impact**: O(n) complexity where n = total connected clients, causing unnecessary network traffic and CPU usage.
**Solution**: Implement topic-based routing or user-specific channels.

### 2. Incomplete Resource Cleanup (Frontend)
**Location**: `frontend/src/pages/Dashboard.jsx` WebSocket and interval handling
**Problem**: Multiple intervals and timeouts created without guaranteed cleanup in all code paths.
**Examples**:
- Reconnection timeout not cleared in all error paths
- WebSocket instances may not be properly closed on component unmount
**Impact**: Memory leaks and potential connection exhaustion over time.

### 3. Database Query Optimization Opportunities
**Location**: Multiple endpoints in `backend/src/app.js`
**Problem**: Several endpoints run multiple separate queries that could be combined or optimized.
**Examples**:
- KPIs endpoint runs 5 separate queries
- Audit logs endpoint uses LEFT JOIN without confirmed indexing
**Impact**: Increased database load and latency, especially under heavy usage.

### 4. Validation Inconsistencies
**Location**: `backend/src/app.js` simulation endpoint (lines 193-221)
**Problem**: Validates `churn_risk` but not other fields (`user_id`, `usage`, `complaints`, `payment_delay`) that are forwarded to simulation service.
**Impact**: Potential errors in downstream services when required fields are missing.

## Detailed Recommendations

### Backend Improvements

#### 1. WebSocket Optimization
- Implement topic-based broadcasting (users subscribe to relevant topics)
- Add message filtering based on user roles/preferences
- Consider using Redis Pub/Sub more effectively for distribution

#### 2. Error Handling Standardization
- Create centralized error handling middleware
- Standardize error response formats
- Add more comprehensive input validation using a library like Joi or express-validator
- Implement proper timeout handling for external service calls

#### 3. Database Performance
- Add composite indexes on frequently queried columns:
  - `agent_memory(user_id, executed_at)`
  - `churn_predictions(user_id, predicted_at)`
  - `retention_actions(user_id, executed_at)`
- Consider query optimization for KPIs endpoint using materialized views or cached aggregations
- Add query timeouts to prevent long-running queries from consuming resources

#### 4. Code Structure Improvements
- Extract business logic into service layers
- Create reusable validation functions
- Implement proper logging correlation IDs across services
- Add request rate limiting to prevent abuse

### Frontend Improvements

#### 1. State Management Optimization
- Reduce prop drilling by implementing Context API or state management library
- Optimize re-renders with better useMemo/useCallback dependencies
- Consider virtual scrolling for large lists (already partially implemented)

#### 2. WebSocket Connection Management
- Implement proper cleanup in useEffect return functions
- Add connection status indicators and retry limits
- Implement message queuing for offline scenarios

#### 3. Performance Optimizations
- Optimize force graph rendering by reducing node/link rebuild frequency
- Implement canvas-based rendering for large datasets
- Lazy load non-critical components
- Optimize image and asset loading

#### 4. Code Quality
- Break down large components into smaller, reusable units
- Standardize error boundaries and loading states
- Add TypeScript for better type safety (if not already planned)

### Database Improvements

#### 1. Indexing Strategy
- Add indexes on foreign keys and frequently filtered columns
- Consider partial indexes for common query patterns
- Regularly analyze query performance with EXPLAIN

#### 2. Connection Pool Tuning
- Adjust pool size based on expected concurrent load
- Monitor and adjust idle/connection timeouts
- Consider using connection pooling at the application level (already implemented)

#### 3. Backup and Maintenance
- Implement regular backup schedules
- Add database health checks
- Consider read replicas for read-heavy workloads

## Security Enhancements

### 1. Input Validation

### 2. Authentication & Authorization
